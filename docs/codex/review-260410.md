# ForgeDesk Technical Review

Date: 2026-04-10

## Scope

Reviewed the Go web application, authentication/session flows, middleware, multi-tenant data access, file handling, assistant/websocket paths, storage, deployment manifests, and current Go test status.

## Executive Summary

The project has a solid baseline in a few important areas: Argon2id password hashing, AES-GCM secret storage, mandatory 2FA, CSRF protection, parameterized SQL, and a restrictive Kubernetes container security context. The largest risks are not in crypto primitives but in authorization consistency and file-serving behavior.

The most serious issue is that arbitrary attachments are publicly served from the application origin with attacker-controlled `Content-Type`, which enables same-origin HTML/JS execution. Separately, ticket, comment, reaction, and ticket-creation paths do not enforce tenant-scoped authorization consistently, so authenticated users can mutate resources outside their org/project boundary if they know IDs. The current test suite is also not green, which lowers confidence in future changes.

## Findings

### 1. Critical: Arbitrary attachment upload can become same-origin stored XSS

Severity: Critical

Evidence:

- `internal/handlers/files.go:37-55` serves any `/files/*` object publicly and reflects stored `Content-Type` directly.
- `internal/handlers/files.go:184-255` allows general attachment uploads of any type and trusts the client-supplied `Content-Type`.
- `templates/pages/ticket_detail.html:148-157` renders attachments as direct links opened in a new tab on the same origin.
- `internal/models/queries.go:1468-1479` persists the attacker-controlled metadata.

Impact:

- Any authenticated user who can upload an attachment can store HTML, SVG, or script-bearing content and have it execute under `smart.madalin.me`.
- Because the content is same-origin, this can access session-authenticated endpoints, CSRF tokens exposed in rendered pages, and other user data visible to that origin.
- `X-Content-Type-Options: nosniff` does not mitigate this because the application is explicitly serving attacker-controlled `Content-Type`.

Recommendation:

- Do not serve untrusted attachments from the application origin.
- Move attachments to a separate download domain or force download with `Content-Disposition: attachment`.
- Derive server-side content type from trusted sniffing or a strict allowlist, not from request headers.
- Restrict attachable types if inline rendering is required.

### 2. High: Broken object-level authorization on ticket, comment, reaction, and ticket-creation endpoints

Severity: High

Evidence:

- `internal/handlers/tickets.go:58-79` correctly uses `GetTicketScoped` for client ticket reads.
- `internal/models/queries.go:1593-1610` already provides an org-scoped ticket lookup.
- `internal/handlers/tickets.go:124-221` creates tickets from arbitrary `project_id` without checking user membership in that project/org.
- `internal/handlers/tickets.go:224-295` updates ticket fields by raw `ticketID` without checking tenant access.
- `internal/handlers/comments.go:22-49` posts comments to arbitrary `ticketID` without checking tenant access.
- `internal/handlers/reactions.go:31-60` toggles reactions on arbitrary ticket/comment IDs without checking existence or tenant access.
- `templates/pages/project_features.html:13`, `templates/pages/project_bugs.html:13`, and `templates/pages/ticket_detail.html:179` expose project IDs in forms, which reduces the practical difficulty of abusing the missing create-ticket check.

Impact:

- A client user can mutate records outside the currently selected org/project boundary if they know or obtain IDs.
- This is a classic BOLA/IDOR class issue in a multi-tenant system.
- The presence of `GetTicketScoped` shows the codebase already knows the correct pattern, but it is not applied consistently on state-changing paths.

Recommendation:

- Centralize authorization checks for all ticket/comment/reaction mutations.
- Resolve target resource -> project -> org and verify membership before every write.
- Validate that `parent_id` belongs to the same project as the new ticket.
- Add negative authorization tests for cross-org mutation attempts.

### 3. Medium: Assistant `update_ticket` claims to update status, but the DB write path never persists it

Severity: Medium

Evidence:

- `internal/handlers/assistant.go:796-838` accepts `status`, records `"status"` in the change list, and emits a status-change activity.
- `internal/models/queries.go:864-871` updates title, description, priority, dates, and assignee, but not `status`.
- `internal/models/queries.go:846-852` has a separate `UpdateTicketStatus` query, which is not used by the assistant `update_ticket` flow.

Impact:

- Assistant responses can claim that status changed when it did not.
- Audit activity can become misleading because the status-change event is recorded even when the `tickets.status` column is unchanged.

Recommendation:

- Either remove `status` from assistant `update_ticket`, or persist it explicitly.
- Add a regression test covering assistant-driven status updates.

### 4. Medium: Rate limiting and session audit fields trust spoofable `X-Forwarded-For`

Severity: Medium

Evidence:

- `internal/middleware/ratelimit.go:63-78` uses the raw `X-Forwarded-For` header as the limiter key.
- `internal/auth/session.go:48-56` stores the raw `X-Forwarded-For` value in the session audit record.

Impact:

- An attacker can rotate or forge `X-Forwarded-For` values to dilute login/2FA rate limiting.
- Session IP logging becomes untrustworthy and can contain full forwarded chains rather than a validated client IP.

Recommendation:

- Trust proxy headers only when requests come from known reverse proxies.
- Parse `X-Forwarded-For` correctly and select the intended client hop.
- Otherwise use `RemoteAddr` after host/port parsing.

### 5. Medium: Attachment deletion does not remove the underlying object, so “deleted” files remain reachable

Severity: Medium

Evidence:

- `internal/handlers/files.go:258-305` deletes only the attachment row.
- `internal/models/queries.go:1502-1507` deletes only from `ticket_attachments`.
- No S3 delete exists in `internal/storage/s3.go`.

Impact:

- Deleting an attachment from the UI does not actually revoke access if someone still has the `/files/...` URL.
- This is a privacy problem in addition to a storage leak.

Recommendation:

- Add object deletion in storage and call it during attachment removal.
- If retention is intentional, mark it explicitly and stop presenting the action as delete.

### 6. Medium: Security header posture is incomplete, and ingress does not compensate

Severity: Medium

Evidence:

- `internal/middleware/securityheaders.go:6-13` sets only a minimal header set.
- No `Content-Security-Policy` or `Strict-Transport-Security` is configured there.
- `deploy/k8s/ingress.yaml` enables TLS but does not add security-header annotations.
- `internal/render/render.go:74-123` shows the app correctly sanitizes markdown, but file serving and future template changes still rely heavily on defense-in-depth that is currently absent.

Impact:

- A CSP would materially reduce the blast radius of any HTML/script injection path.
- HSTS would harden transport guarantees for a TLS-only deployment.

Recommendation:

- Add a strict CSP appropriate for HTMX/Alpine usage.
- Add HSTS at ingress or app level once HTTPS-only deployment is guaranteed.

### 7. Low: The default verification path is currently broken

Severity: Low

Evidence:

- `go test ./...` fails as of 2026-04-10.
- `internal/ai/gemini_test.go:9-35` still expects 6 tool declarations, while `internal/ai/gemini.go:69-210` now defines 15.
- `internal/ai/gemini_test.go:56-85` still requires `create_ticket.title`, while `internal/ai/gemini.go:96-110` intentionally allows title omission.
- `internal/config/config.go:131-137` now requires `AES_ENCRYPTION_KEY`, but `internal/config/config_test.go:7-12` does not set it in shared test setup, causing multiple config tests to fail.

Impact:

- The project no longer has a trustworthy green baseline.
- CI/test failures will mask new regressions and slow down review.

Recommendation:

- Update the stale tests immediately.
- Treat green unit tests as a release gate before expanding functionality further.

## Positive Notes

- Authentication fundamentals are better than average: Argon2id password hashing, mandatory TOTP/recovery-code flow, and AES-GCM encryption for stored secrets.
- CSRF protection is wired into HTML routes and uses strict cookie settings.
- Markdown rendering is sanitized with Bluemonday before being emitted.
- The Kubernetes deployment drops Linux capabilities, runs as non-root, disables privilege escalation, and uses a read-only root filesystem.

## Verification

Executed:

- `go test ./...`

Result:

- Failed in `internal/ai` because tests no longer match the expanded assistant tool surface.
- Failed in `internal/config` because tests do not provide the now-required `AES_ENCRYPTION_KEY`.
- Other reviewed packages were green in this run, including `internal/auth`, `internal/handlers`, `internal/middleware`, `internal/models`, and `internal/ws`.

## Recommended Fix Order

1. Fix attachment serving so untrusted files cannot execute under the app origin.
2. Apply tenant authorization checks to every state-changing ticket/comment/reaction endpoint.
3. Repair assistant `update_ticket` status persistence.
4. Correct proxy-aware IP handling for rate limiting and session logging.
5. Make attachment deletion actually remove or revoke stored objects.
6. Restore a green test suite and add cross-tenant negative tests.
